EE108B - Programming Assignment #1

Due: Thursday, Feb 8th

Goals:

This assignment is intended to familiarize you with the SPIM environment for later assignments; to test your understanding of pointer implementations and procedure call conventions.

Your task:

Implement quicksort using the MIPS instruction set outlined in the SPIM documentation and in lecture.

Look at quicksort.c for a C version of the quicksort routine. Your job is to add the required assembly code to the skeleton file, quicksort.s, such that the assembly code has the same functionality as the C code. The skeleton file can be copied to your local directory:

cp /usr/class/ee108b/WWW/pas/PA1/quicksort.s .

Comments within quicksort.s clearly specify the functionality that needs to be added.

Guidelines:

  • Read Chapter 3 (3.1 - 3.11) before you start. Be sure to follow the procedure call convention detailed in Appendix A.6. You may also find Appendix A.9 and A.10 very helpful.
  • Make sure that all function arguments are passed according to the established convention! The arguments shall be passed in the following order: pointer to the first element of the array, and num_of_elements.  
  • Follow the register and procedure call conventions (i.e., which registers are intended for what use and the sequence of steps in a procedure call). Appendix A of the textbook is a good reference in this respect. Compilers have to follow the conventions so you should, too.

Hints:

  • Be sure you have a good understanding of how a procedure call works before you attempt to code.
  • The assignment is reasonably straightforward and shouldn't take too much more than 80 or 90 lines of assembly. If you have much more than that then maybe you should look at it again.
  • You can work on any platform you like, but it is your responsibility to make sure your project will run on the leland workstations, which is the only platform we support.

Requirements:

  • By 11:59 PM of the due date, you should electronically submit your project. The directory you submit must have a README file and your quicksort.s
    Please see the submission instructions for details.
  • The TAs will be testing your code using the spim that runs on elaine or myth machines. Use spimsal and other platforms at your own risk. It is your responsibility to make sure that your code runs correctly on the Sparc stations at Sweet Hall.
  • Add code to your local copy of quicksort.s, but DO NOT CHANGE any of the ORIGINAL quicksort.s code. If you think there's a bug, please email the TAs before you make any changes.
  • We should be able to run your program by issuing the command

spim -file quicksort.s < inputfile > outputfile

where inputfile is a file representing an array of integers. The first number in the file specifies the number of integers in the array. The subsequent integers are the actual array elements. For example, this inputfile should generate this outputfile with the above command.

Notes on Grading:

  •  We will be using an automatic script, so the format of your output must exactly match ours (you can figure out the format from the sampe output files).
  • Coding efficiency will not be graded (ie total number of instructions executed, instructions per loop, efficient memory usage, etc.). All we are looking for is functionality. However, you do need to implement your code in the exact same way as it is implemented in the quicksort.c file, meaning when there is a recursive call, you must make a recursive call, when there is dynamically allocated memory, you must dynamically allocate memory, etc. Also, use reasonable formatting and white space usage in your code.
  • We have provided you a sample input and output file. These will not be the only test cases used for grading, so you may want to test your code a bit more than with just those test cases. However, we will not be trying to break your code with ridiculous inputs, such as huge numbers, huge lists of numbers that could cause your stack and heap to collide.

 

Good luck and happy sorting!
